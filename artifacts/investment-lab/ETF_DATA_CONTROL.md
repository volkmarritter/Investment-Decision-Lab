# Controlling ETF Data Updates

> **Audience:** anyone who maintains the Investment Decision Lab and needs to know where ETF numbers come from, how they get refreshed, and what to do when something is wrong.
>
> **Companion docs:**
> - `DOCUMENTATION.md` — full functional / engine logic.

## 1. The big picture

The Investment Decision Lab is **frontend-only**. There is no backend database and no live pricing call from the browser — everything users see is baked into the bundle at build time.

What gets shown to users for any given ETF is the result of merging two layers:

| Layer | File(s) | Editable how? |
|---|---|---|
| **Curated catalog** (editorial source of truth) | `src/lib/etfs.ts`, `src/lib/lookthrough.ts` | Hand-edit in source, commit |
| **Scraped overrides** (auto-refreshed) | `src/data/etfs.overrides.json`, `src/data/lookthrough.overrides.json` | Generated by scripts; committed to git |

At module load, `etfs.ts` and `lookthrough.ts` shallow-merge the override JSON on top of the curated catalog. When the override file is empty (the default committed state) the engine behaves exactly like the curated catalog alone.

```
┌────────────────────────────┐      ┌──────────────────────────────────┐
│ src/lib/etfs.ts (CATALOG)  │ ◄────┤ shallow-merge at module load     │
│ src/lib/lookthrough.ts     │      │ (overrides win field-by-field)   │
└────────────────────────────┘      └─────────────────▲────────────────┘
                                                      │
                          ┌───────────────────────────┴───────────────────────────┐
                          │                                                       │
                ┌─────────┴─────────┐                                   ┌─────────┴─────────┐
                │ etfs.overrides    │                                   │ lookthrough.over- │
                │ .json             │                                   │ rides.json        │
                └─────────▲─────────┘                                   └─────────▲─────────┘
                          │ written by                                            │ written by
                ┌─────────┴────────────────────┐                       ┌─────────┴─────────┐
                │ scripts/refresh-justetf.mjs  │                       │ scripts/refresh-  │
                │   (core fields + listings)   │                       │ lookthrough.mjs   │
                └─────────▲────────────────────┘                       └─────────▲─────────┘
                          │                                                       │
                          └────────── scrapes justETF.com HTML ───────────────────┘
```

## 2. What is curated vs. what is scraped

Knowing this matters: **hand-edits to scraped fields will be overwritten on the next refresh.** If you need a permanent change to a scraped field, change the source (curate) or extend the scraper.

### Curated (you control by editing `src/lib/etfs.ts`)

- `name`, `isin`, `domicile`
- `currency` (the fund's base currency)
- `defaultExchange`
- editorial comments
- The currency mix for **hedged ETFs** (kept authoritative on purpose; the set is `HEDGED_ISINS` in the scraper)

### Scraped (do not hand-edit — refreshed automatically)

From `refresh-justetf.mjs` into `etfs.overrides.json`:
- `terBps` (Total Expense Ratio in basis points)
- `aumMillionsEUR`
- `inceptionDate`
- `distribution` (Acc / Dist)
- `replication` (Physical / Synthetic)
- `listings` (per-exchange ticker map: LSE, XETRA, SIX, Euronext)

From `refresh-lookthrough.mjs` into `lookthrough.overrides.json`:
- `topHoldings` (top-10 positions)
- `geo` (geographic breakdown)
- `sector` (sector breakdown)

## 3. The automatic schedule

Three GitHub Actions workflows commit fresh JSON to the repo on a cadence:

| Cadence | What | Workflow file |
|---|---|---|
| **Nightly, 02:00 UTC** | Listings (LSE / XETRA / SIX / Euronext tickers) | `.github/workflows/refresh-listings.yml` |
| **Weekly, Sunday 03:00 UTC** | Core fund metadata (TER, AUM, inception, distribution, replication) | `.github/workflows/refresh-data.yml` |
| **Monthly, 1st at 04:00 UTC** | Look-through (top-10 holdings, geo, sector) | `.github/workflows/refresh-lookthrough.yml` |

Each workflow runs the matching scraper, commits the changed JSON files back to `main`, and pushes. The next deploy picks up the new data — no manual rebuild needed.

You can also **trigger any of them on demand** from the GitHub **Actions** tab via the *Run workflow* button.

## 4. Manual refresh — CLI commands

All commands are run from `artifacts/investment-lab/`.

### Core fields (TER, AUM, listings, distribution, replication)

```bash
# Refresh everything in one go
node scripts/refresh-justetf.mjs

# Only the core editorial fields (TER, AUM, inception, distribution, replication)
node scripts/refresh-justetf.mjs --mode=core

# Only the per-exchange listings
node scripts/refresh-justetf.mjs --mode=listings

# Only one specific ETF (by ISIN)
node scripts/refresh-justetf.mjs IE00B4L5Y983

# Combine: refresh listings for a single ETF
node scripts/refresh-justetf.mjs --mode=listings IE00B4L5Y983

# Preview what would change without writing the file
DRY_RUN=1 node scripts/refresh-justetf.mjs
```

### Look-through (top-10 holdings, geo, sector)

```bash
# Refresh holdings + breakdowns for the whole catalog
node scripts/refresh-lookthrough.mjs

# Just one ETF
node scripts/refresh-lookthrough.mjs IE00B4L5Y983

# Preview only
DRY_RUN=1 node scripts/refresh-lookthrough.mjs
```

After running, commit the changed `src/data/*.overrides.json` files. The frontend will pick up the new values on the next build.

## 5. How to add or remove an ETF

1. **Add or remove it in `src/lib/etfs.ts`** — at minimum the entry needs `isin`, `name`, `domicile`, `currency`, `defaultExchange`. If it has special look-through behaviour, also add an entry in `src/lib/lookthrough.ts`.
2. **Backfill scraped data:**
   ```bash
   node scripts/refresh-justetf.mjs <NEW_ISIN>
   node scripts/refresh-lookthrough.mjs <NEW_ISIN>
   ```
3. **Commit** both the source change and the regenerated `src/data/*.overrides.json` files.
4. **Deploy** as usual.

For a removed ETF, after deleting it from `src/lib/etfs.ts`, run a full `node scripts/refresh-justetf.mjs` so the override file no longer carries a stale entry.

## 6. How users see freshness

Each override file carries a `_meta` block:

```json
{
  "_meta": {
    "lastRefreshed": "2026-04-20T03:00:14Z",
    "lastRefreshedMode": "core",
    "perIsin": {
      "IE00B4L5Y983": { "topHoldingsAsOf": "2026-04-01T04:00:00Z" }
    }
  },
  "IE00B4L5Y983": { "terBps": 7, "aumMillionsEUR": 73824, "...": "..." }
}
```

The component `src/components/investment/SnapshotFreshness.tsx` reads this metadata and surfaces a "Data as of …" hint to the user, so the UI is always honest about how recently the numbers were verified.

## 7. Common operational tasks

| Task | Command / step |
|---|---|
| One ETF's TER looks wrong | `node scripts/refresh-justetf.mjs --mode=core <ISIN>`, inspect the diff, commit |
| Listings stale on one exchange | `node scripts/refresh-justetf.mjs --mode=listings <ISIN>` |
| Holdings out of date | `node scripts/refresh-lookthrough.mjs <ISIN>` |
| You want to see what *would* change | Prefix any command with `DRY_RUN=1` |
| GitHub Action failed overnight | Re-run from the Actions tab; failures usually mean justETF changed its HTML — check `scripts/refresh-justetf.mjs` parsers |
| Override stuck on a stale value | Open the relevant `*.overrides.json`, delete the entry, re-run the scraper |
| Lock a value against future scrapes | Move the field out of "scraped" into the curated `CATALOG` (and remove it from the scraper writer) |

## 8. Future options (not implemented today)

If the maintenance flow gets heavier, three add-ons are easy to bolt on later:

1. **In-app "Refresh now" button** — a small admin endpoint on the api-server that triggers a scraper run; surfaced via a hidden control in the Lab.
2. **Manual override layer** above the scraper — a `etfs.manual.json` whose values win over the scraped overrides, so individual fields can be pinned without modifying the curated source.
3. **Per-ETF "last verified" badge** in the ETF table itself, for finer-grained transparency than the current global freshness banner.

Open a request when any of these become useful.
